# Kalman Filter Hello World

这个目录先不强调“控制器升级”，而强调“状态估计”。  
你现在已经做过 `PID` 和 `LQR`，下一步最需要补上的能力，是：

- 只有位置测量时，怎么估计速度
- 测量有噪声时，怎么得到更平滑、更可信的状态

这正是 `Kalman Filter` 的典型用途。

---

## 1. 这个脚本到底在解决什么问题

脚本 `hello_kalman.py` 只做一件很基础但很关键的事：

- 真实系统里有位置 `x_true` 和速度 `v_true`
- 传感器只能测到带噪声的位置 `x_meas`
- 滤波器要根据“模型 + 测量”去实时估计：
  - `x_hat`：估计位置
  - `v_hat`：估计速度

换句话说，这不是在直接研究“怎么控得更好”，而是在研究：

- 我明明没测到速度，能不能把速度估出来
- 我测到的位置很抖，能不能估出一个更接近真实值的位置

这件事对后面几乎所有模型驱动控制都很关键，因为：

- `LQR` 默认你知道完整状态
- `LQG` 本质上就是 `LQR + Kalman`
- `MPC` 也通常更希望拿到完整状态

如果状态不完整，控制器再漂亮，落地时也会很别扭。

---

## 2. 先建立一个直觉：Kalman Filter 不是“神奇平均器”

初学者最容易把它理解成“把噪声做平滑”。  
这个理解不完全错，但不够准确。

更准确地说，Kalman Filter 每一步都在做两件事：

1. **先根据模型预测下一步会在哪里**
2. **再根据传感器测量修正这个预测**

所以它不是只看测量值，也不是只看模型，而是在动态权衡：

- 如果模型更可信，就多信模型
- 如果测量更可信，就多信测量

这就是后面 `Q` 和 `R` 两个参数的物理意义。

---

## 3. 脚本里的状态、输入、测量分别是什么

这个例子使用一个 1D 的最小模型：

- 状态：`[x, v]`
  - `x` 是位置
  - `v` 是速度
- 输入：`a_cmd`
  - 你可以把它理解成“系统认为自己施加的加速度命令”
- 测量：`z`
  - 这里只有位置测量

对应的离散状态空间模型是：

`x(k+1) = F x(k) + B u(k) + w(k)`

`z(k) = H x(k) + v(k)`

其中：

- `F`：状态如何自然演化
- `B`：控制输入如何影响状态
- `H`：传感器能从状态里读出什么
- `w(k)`：过程噪声，表示模型不完美
- `v(k)`：测量噪声，表示传感器有误差

这个目录里只测位置，所以 `H` 只把位置那一维取出来。

---

## 4. 先看脚本前半段：模型和噪声是怎么定义的

脚本开头这一段：

```python
dt = 0.01
T = 6.0

disturbance = 0.15
measurement_sigma = 0.01
process_sigma = 0.02
```

可以这样理解：

- `dt`：采样周期，每 `0.01s` 更新一次
- `T`：总仿真时间 `6s`
- `disturbance`：真实系统里一直存在的小外扰
- `measurement_sigma`：位置测量噪声标准差
- `process_sigma`：过程噪声标准差

这里最重要的不是数值本身，而是你要建立这个概念：

- **真实系统不会完全等于模型**
- **传感器也不会给你完美真值**

Kalman Filter 就是为这种现实情况设计的。

---

## 5. 为什么 `F`、`B` 要写成现在这样

脚本里有这几行：

```python
F = np.array([
    [1.0, dt],
    [0.0, 1.0],
], dtype=float)

B = np.array([
    [0.5 * dt * dt],
    [dt],
], dtype=float)
```

这其实就是把最基础的运动学离散化：

- `x_next = x + v * dt + 0.5 * a * dt^2`
- `v_next = v + a * dt`

所以：

- `F` 负责描述“如果没有输入，状态自己怎么往下走”
- `B` 负责描述“加速度输入会怎么改变位置和速度”

如果你已经学过 `LQR`，这里可以把它看成一个更简单的状态空间模型。  
区别是这次重点不是设计反馈增益，而是用这个模型辅助估计状态。

---

## 6. 为什么 `H = [[1, 0]]`

脚本里：

```python
H = np.array([[1.0, 0.0]], dtype=float)
```

这表示传感器只能看到状态向量 `[x, v]` 里的第一项，也就是位置 `x`。

所以这里的设定就是：

- 位置可测
- 速度不可测

这正好对应你后面做球板/平台时很常见的情况：

- 相机能看到位置
- 速度通常不是直接测来的，而是估计出来的

---

## 7. `Q` 和 `R` 到底是什么

脚本里：

```python
Q = np.array([
    [1e-5, 0.0],
    [0.0, process_sigma ** 2],
], dtype=float)
R = np.array([[measurement_sigma ** 2]], dtype=float)
```

可以把它们先理解成“你对世界不确定性的描述”：

- `Q`：过程噪声协方差，表示你觉得模型有多不准
- `R`：测量噪声协方差，表示你觉得传感器有多吵

初学时只记住这个直觉就够了：

- `Q` 变大：更不信模型，更愿意让测量来纠正
- `R` 变大：更不信测量，更愿意按模型往前推

你以后调 Kalman，最先感受到的就是这两个参数带来的权衡。

---

## 8. 真值、测量值、估计值三者分别是什么

脚本里有三套“状态”概念，不要混：

### 8.1 真值

```python
x_true = 0.05
v_true = 0.0
```

这代表仿真器里的真实系统状态。  
现实里你通常拿不到这个真值，它只是为了让我们验证滤波器效果。

### 8.2 测量值

```python
z = x_true + rng.normal(0.0, measurement_sigma)
```

这表示传感器读到的值。  
它等于真实位置加上一点测量噪声，所以会抖。

### 8.3 估计值

```python
x_hat = np.array([[0.0], [0.0]], dtype=float)
P = np.eye(2) * 0.1
```

其中：

- `x_hat` 是滤波器当前对状态的估计
- `P` 是估计误差协方差，表示“我对当前估计有多确定”

`P` 大表示不太确定，`P` 小表示更有把握。

---

## 9. 主循环到底在做什么

主循环里每一步都分成三层逻辑：

1. 真实系统往前走一步
2. 传感器给一个带噪声的位置测量
3. Kalman Filter 做一次预测和更新

下面按脚本顺序拆开看。

### 9.1 先让真实系统运行

```python
a_cmd = -0.8 * x_true - 0.3 * v_true

process_noise = rng.normal(0.0, process_sigma)
a_true = a_cmd + disturbance + process_noise
v_true += a_true * dt
x_true += v_true * dt
```

这里不是在强调控制器设计，而只是为了让系统“动起来”，这样滤波器才有东西可估。

- `a_cmd` 是一个很简单的开环/弱反馈式激励
- `disturbance` 是恒定外扰
- `process_noise` 是随机过程噪声
- `a_true` 是真实系统实际受到的加速度

注意这个细节很重要：

- 滤波器在预测时只知道 `a_cmd`
- 但真实系统实际经历的是 `a_true`

两者之间有偏差，正是现实世界里“模型不完美”的来源。

### 9.2 再生成传感器测量

```python
z = x_true + rng.normal(0.0, measurement_sigma)
```

这一步表示传感器只给位置，而且有噪声。  
所以你看到的 `z` 并不等于 `x_true`。

### 9.3 Predict：先按模型往前推

```python
x_hat = F @ x_hat + B * a_cmd
P = F @ P @ F.T + Q
```

这一步回答的问题是：

- “如果先不看传感器，只按照模型推演，下一步状态应该是什么？”

两条式子的意义分别是：

- `x_hat = F @ x_hat + B * a_cmd`
  - 预测下一时刻的状态估计
- `P = F @ P @ F.T + Q`
  - 预测下一时刻的不确定性

为什么 `P` 会加上 `Q`？

- 因为你往前推一步以后，通常不会比之前更确定
- 模型本身有误差，所以不确定性会增加

### 9.4 Update：再用测量把估计拉回来

```python
y = np.array([[z]], dtype=float) - H @ x_hat
S = H @ P @ H.T + R
K = P @ H.T @ np.linalg.inv(S)
x_hat = x_hat + K @ y
P = (np.eye(2) - K @ H) @ P
```

这一段是 Kalman Filter 的核心。

#### `y`：创新 / 残差

`y = z - H @ x_hat`

它表示：

- 传感器看到的值
- 减去
- 你根据模型预测传感器应该看到的值

如果 `y` 很大，说明预测和测量差很多，就需要更多修正。

#### `S`：残差的不确定性

`S = H P H^T + R`

可以理解成：

- 当前预测不确定性
- 加上传感器测量不确定性

#### `K`：Kalman 增益

`K = P H^T S^-1`

这一步很关键。  
`K` 决定“这次修正时到底该信测量多少”。

你可以这样记：

- `K` 大：修正幅度大，更信测量
- `K` 小：修正幅度小，更信模型

#### 更新状态和协方差

`x_hat = x_hat + K @ y`

表示把预测值按残差方向修正。

`P = (I - K H) @ P`

表示融合了这次测量之后，通常会比刚预测完更确定，所以不确定性下降。

## 10. 为什么它能估计出速度

这是初学者最关心的问题之一：

- 明明没有速度传感器
- 为什么 `v_hat` 还能出现

原因是速度虽然没被直接测量，但它会影响位置随时间的变化。  
Kalman Filter 利用的是：

- 状态之间的动力学耦合
- 连续时刻的位置测量

如果系统模型写得合理，位置的变化趋势里就已经包含了速度信息。  
滤波器会把这些信息“倒推出”速度估计。

这也是为什么状态空间方法这么重要：  
它让不同状态之间的关系可以被系统性利用。

## 11. 运行时应该重点看什么

运行：

```bash
python 03_helloworld_kalman/hello_kalman.py
```

输出里重点看这几组量：

- `x_true`：真实位置
- `x_meas`：带噪声的位置测量
- `x_hat`：滤波器估计位置
- `v_true`：真实速度
- `v_hat`：滤波器估计速度

正常现象通常是：

- `x_meas` 比较抖
- `x_hat` 比 `x_meas` 更平滑，也更接近 `x_true`
- `v_hat` 不会完全等于 `v_true`，但会逐渐跟住趋势

如果 `x_hat` 很平滑但反应特别迟钝，常见原因是：

- `R` 设太小或太大导致权衡不合理
- `Q` 太保守，滤波器过度相信模型

## 12. 你可以立刻做的三个小实验

### 实验 1：把测量噪声变大

把：

```python
measurement_sigma = 0.01
```

改大，比如改成 `0.05`。

观察：

- `x_meas` 会更抖
- `x_hat` 仍然会尽量保持稳定

这样你会更直观看到滤波器的价值。

### 实验 2：把 `R` 调大

让滤波器更不信测量。

你会看到：

- `x_hat` 更平滑
- 但有时会更慢地贴近真实变化

### 实验 3：把 `Q` 调大

让滤波器承认“模型没那么准”。

你会看到：

- 滤波器更愿意跟着测量修正
- 响应可能更快，但也可能更敏感

这三个实验做完，你对 Kalman 的调参直觉会明显更清楚。

## 13. 它和你已经学过的 PID、LQR 是什么关系

- `PID`：主要解决“怎么根据误差直接出控制量”
- `LQR`：主要解决“如果完整状态已知，怎么最优反馈”
- `Kalman`：主要解决“如果状态不完整、测量有噪声，怎么估计状态”

所以它们不是互相替代，而是不同层次的问题。

你当前这条学习路线很合理：

1. 先理解 `PID`
2. 再理解 `LQR`
3. 然后补上 `Kalman`
4. 最后组合成 `LQG`

这样逻辑是连贯的。

## 14. 对未来 2 自由度平台的直接意义

未来上 2D 平台时，你很可能会变成：

- 状态：`[x, vx, y, vy]`
- 测量：相机给出 `(x, y)`
- 估计：通过滤波器补出 `(vx, vy)`

那时本质没有变，只是从 1D 变成 2D：

- 现在你学的是“只测 `x`，估计 `v`”
- 以后会变成“只测 `x, y`，估计 `vx, vy`”

所以这个 hello world 虽然小，但它是后面所有“估计 + 控制”链路的基础。
