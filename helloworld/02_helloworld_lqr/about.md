# Ball-on-Beam LQR 入门讲解（含可运行示例）

这个目录给出一个和 `01_helloworld_pid` 对应的 LQR 版本：同样是最小可运行仿真，但控制器从“经验调参的双环 PID”换成“基于状态空间模型的最优状态反馈”。

---

## 1. 这个问题到底在控什么？

目标仍然一样：

- 让小球停在杆中心，也就是 `x -> 0`
- 控制输入不是直接改位置，而是改杆角度 `theta`
- 杆角度改变后，小球产生加速度，再经过积分变成速度和位置变化

这也是这个系统难控的根源：

- 你想管的是位置
- 你真正能下手的是杆角度
- 角度对位置的影响中间还隔着速度这一层

所以它天然是一个多状态系统，不只是“一个误差输入，一个输出结束”。

---

## 2. 为什么这里适合用 LQR？

PID 的思路是：

- 看误差
- 按比例、积分、微分去修正

LQR 的思路是：

- 先把系统写成状态空间模型
- 明确告诉控制器“哪些状态偏差更不能忍、控制动作多大算贵”
- 让它自动算出一个反馈增益矩阵 `K`

然后实时控制时只做一件事：

`u = -Kx`

也就是：

- 位置偏了，打一点控制
- 速度太大，也打一点控制
- 杆角偏了，也打一点控制
- 杆角速度太大，也打一点控制

不是分层去想，而是一次性综合考虑所有状态。

这对球杆系统很合适，因为它本来就是一个典型的多状态线性控制问题。

---

## 3. 本示例用了什么状态？

脚本里把状态定义为：

`[x, v, theta, theta_dot]`

分别代表：

- `x`：小球位置
- `v`：小球速度
- `theta`：杆角度
- `theta_dot`：杆角速度

控制输入 `u` 定义为：

- 电机角速度命令

注意这里不是“电机电压”或者“电机力矩”，而是一个已经简化过的输入。这样做是为了让入门代码尽量短，同时又保留足够真实的动态特性。

---

## 4. 模型是怎么来的？

在小角度近似下：

### 小球部分

`x_ddot = g * theta - damping * v`

含义：

- 杆向右倾，小球就向右加速
- 速度越大，阻尼项越强，能抑制来回摆动

### 电机部分

脚本用了一个一阶惯性近似：

`theta_ddot = -(1 / tau) * theta_dot + (1 / tau) * u`

它对应代码里的更新方式：

- `theta_dot` 逐步逼近 `u`
- `theta` 再由 `theta_dot` 积分得到

这个模型比“`theta = u` 直接成立”更合理，也更利于看出控制器为什么稳定。

---

## 5. 为什么能写成状态空间？

因为上面的动态方程都能整理成：

`x_dot = A x + B u`

这里的状态向量不是单个位置 `x`，而是整个四维状态：

`state = [x, v, theta, theta_dot]^T`

在 `hello_lqr.py` 里，连续时间矩阵是：

```python
A = [
    [0, 1, 0, 0],
    [0, -damping, g, 0],
    [0, 0, 0, 1],
    [0, 0, 0, -1/beam_tau],
]

B = [
    [0],
    [0],
    [0],
    [1/beam_tau],
]
```

读法很直接：

- 第一行表示 `x_dot = v`
- 第二行表示 `v_dot = -damping * v + g * theta`
- 第三行表示 `theta_dot = theta_dot`
- 第四行表示电机的一阶响应

---

## 6. 为什么脚本里还要离散化？

因为仿真是按时间步长 `dt` 一步一步更新的，所以控制器也要基于离散模型设计。

脚本里用了最简单的前向欧拉离散化：

- `Ad = I + A * dt`
- `Bd = B * dt`

这不是最精确的方法，但对这个入门 demo 足够，而且便于理解。等以后你要做更高精度版本，可以换成矩阵指数离散化。

---

## 7. LQR 到底在优化什么？

LQR 不是“凭感觉找增益”，而是在最小化一个代价函数：

`J = sum(x^T Q x + u^T R u)`

意思是：

- 状态偏差要罚
- 控制动作太猛也要罚

### `Q` 是状态惩罚矩阵

本示例使用：

```python
Q = diag([120, 12, 260, 8])
```

对应直觉：

- 小球位置偏差要重罚
- 速度也要管
- 杆角度偏差也要重罚
- 杆角速度稍微轻一点

### `R` 是控制惩罚矩阵

本示例使用：

```python
R = [[1.5]]
```

它越大，控制器越保守：

- 输出更小
- 动作更柔和
- 收敛一般更慢

它越小，控制器越激进：

- 输出更大
- 回中更快
- 更容易碰到饱和或振荡

---

## 8. Riccati 方程和 `K` 是怎么得到的？

LQR 的核心不是手工猜 `K`，而是先求一个矩阵 `P`，满足离散代数 Riccati 方程（DARE）。

这一步做完后，就能得到：

`K = (R + B^T P B)^(-1) B^T P A`

脚本里没有依赖 `scipy`，而是用迭代方式自己求 `P`：

- 先设 `P = Q`
- 不断重复 Riccati 更新
- 直到前后两次变化足够小

这么做的目的很明确：

- 让示例只依赖 `numpy`
- 用户 clone 下来就能跑
- 也方便你看到 LQR 计算流程到底是什么

---

## 9. 控制循环里实际发生了什么？

每一个仿真步，脚本都做这几件事：

1. 读取当前状态 `state = [x, v, theta, theta_dot]`
2. 计算误差状态 `error_state = state - target`
3. 用 LQR 公式求控制输入 `u = -K * error_state`
4. 把 `u` 限幅，避免命令无限大
5. 更新电机状态 `theta_dot`
6. 更新杆角度 `theta`
7. 更新小球速度 `v`
8. 更新小球位置 `x`

这和 PID 版一样，也是“读状态 -> 算控制 -> 推进仿真”，只是控制律从 PID 换成了矩阵乘法。

---

## 10. 为什么还要做限幅？

因为真实系统一定有限制，仿真里没有限制往往会得到不可信的结果。

本示例做了两种限制：

- 电机命令限幅：`motor_limit`
- 杆角度限幅：`theta_limit`

这些限制的意义是：

- 防止控制器在初始误差较大时给出过大的控制命令
- 让仿真行为更接近真实机构
- 避免“数学上能算，物理上做不到”

要注意一点：

标准 LQR 是在线性、无约束条件下推导出来的。  
加了限幅后，严格意义上它已经不是纯线性系统了，但对这种入门仿真来说，这样做是合理且必要的。

## 11. 如何运行

```bash
python 02_helloworld_lqr/hello_lqr.py
```

你会看到日志大致包含：

- `x`：位置
- `v`：速度
- `theta`：杆角度
- `theta_dot`：杆角速度
- `u`：控制输入

理想现象是：

- `x` 逐渐靠近 0
- `v` 逐渐减小
- `theta` 最终回到接近 0 的小角度
- 控制输入 `u` 在开始较大，随后变小

---

## 12. `Q` 和 `R` 怎么调？

这是 LQR 最关键的入门内容。

### 如果你想更快把球拉回中心

可以尝试：

- 增大 `Q[0,0]`，更重罚位置误差
- 适度增大 `Q[1,1]`，抑制速度
- 适度减小 `R`

代价是：

- 控制更猛
- 更容易打到饱和

### 如果你想让杆动作更温和

可以尝试：

- 增大 `Q[2,2]` 或 `Q[3,3]`
- 增大 `R`

效果通常是：

- 杆不会那么激进
- 收敛可能变慢

### 一个实用的调参逻辑

1. 先保证能稳定
2. 再压超调
3. 最后再追求更快

不要一开始就把 `R` 调得很小，否则你看到的可能不是“更强”，而是“更容易打满限制”。

---

## 13. 它和 PID 版的核心差别是什么？

PID 版的思维方式是：

- 外环看位置
- 内环看角度
- 分两层调

LQR 版的思维方式是：

- 把系统作为一个整体
- 一次性利用全部状态
- 用代价函数决定控制偏好

所以两者的主要差异不是“谁更高级”，而是“建模方式不同”：

- PID 更偏工程经验、容易快速上手
- LQR 更偏模型驱动、结构更统一

如果系统状态可测，模型也比较清楚，LQR 往往会更干净。

---

## 14. 这个 demo 故意省略了什么？

为了让入门版本聚焦控制本身，这里没有加入：

- 传感器噪声
- 状态观测器
- 非线性大角度模型
- 执行器死区
- 摩擦的复杂非线性

这些内容后面都可以加，但不应该一开始就堆进去。先把“状态空间 + LQR + 收敛”这条主线吃透更重要。

## 15. 建议下一步做什么？

1. 把初始位置从 `0.05` 改成 `0.10`，看控制是否依旧稳定。
2. 修改 `Q` 和 `R`，观察“快慢”和“平稳性”的变化。
3. 把 `damping` 降低，模拟更滑的轨道，再重新整定。
4. 给 `x_target` 一个非零常值，尝试做定点保持。
5. 后续再引入 Kalman Filter，做 `LQG` 版本。

---

## 16. 小结

这个示例想传达的重点只有四个：

- 球杆系统很适合用状态空间描述
- LQR 的本质是“状态偏差”和“控制代价”的权衡
- `Q`、`R` 比单纯的 `kp`、`kd` 更接近“设计目标”
- 在入门工程里，适度的限幅和简化模型比盲目追求公式完整更重要
