# ADRC Hello World

这里的 `ADRC` 全名是 `Active Disturbance Rejection Control`，也就是主动抗扰控制。

如果前面的 `PID`、`LQR`、`MPC` 更偏向“先把模型写清楚，再根据模型设计控制器”，那么 `ADRC` 更关心另一类现实问题：

- 模型不准怎么办
- 外扰一直存在怎么办
- 系统里有些未知项根本写不清怎么办

`ADRC` 给出的回答很有工程味：

- 不要执着于把所有未知因素逐个建清楚
- 先把它们打包成“总扰动”
- 再用观测器在线估计这个总扰动
- 最后在控制输入里把它主动补偿掉

这也是为什么 `ADRC` 在实体系统里经常很有吸引力。

---

## 1. 先建立一个直觉：ADRC 到底在做什么

你可以先把 `ADRC` 理解成两层结构：

1. 用一个观测器去估计系统当前状态和总扰动
2. 用一个相对简单的控制器，加上扰动补偿，把系统拉回目标

所以它的味道不是：

- “我已经把世界建模得非常准，所以我能精确控制”

而更像是：

- “世界里肯定有很多没建模的东西，那我就边运行边估出来，再把它抵消”

这就是“主动抗扰”这四个字的核心。

---

## 2. 这个脚本在做什么

脚本 `hello_adrc.py` 用一个最小二阶对象演示 `ADRC` 的基本结构。

被控对象非常简单：

- 状态：位置 `x`、速度 `v`
- 控制输入：`u`
- 未知扰动：`disturbance`

真实系统写成：

`x_ddot = b0 * u + disturbance`

其中：

- `b0` 表示你认为控制输入对系统加速度的大致作用强度
- `disturbance` 是真实系统里的恒定外扰

这个脚本的目标是：

- 就算系统里一直有一个外扰
- 也尽量把位置 `x` 拉回 0

这正是 `ADRC` 想擅长处理的问题。

---

## 3. 什么叫“总扰动”

这是 `ADRC` 最关键的概念。

在很多经典控制方法里，你会倾向于把未知因素拆开：

- 摩擦是多少
- 偏置是多少
- 负载变化是多少
- 参数误差是多少

而 `ADRC` 的思路更直接：

- 不管这些未知因素分别叫什么
- 只要它们最终都会影响系统加速度
- 那就把它们打包成一个总项

在这个示例里，最简单，总扰动就直接由：

- 恒定外扰 `disturbance`

来体现。

但在更真实的系统里，总扰动通常可以包含：

- 外部扰动
- 未建模动态
- 参数不准
- 摩擦变化
- 偏置项

所以“总扰动”不是一个神秘新物理量，它更像一个工程上的统一打包变量。

---

## 4. 这个脚本里的模型怎么理解

脚本开头的关键参数是：

```python
dt = 0.005
T = 5.0
b0 = 9.81
disturbance = 1.5

kp = 45.0
kd = 12.0

beta1 = 120.0
beta2 = 2200.0
beta3 = 8000.0
```

可以这样读：

- `dt`
  - 仿真步长
- `T`
  - 总仿真时间
- `b0`
  - 控制输入对加速度的名义增益
- `disturbance`
  - 真实系统里一直存在的外扰
- `kp`、`kd`
  - 控制器里的 `PD` 增益
- `beta1`、`beta2`、`beta3`
  - `ESO` 的观测器增益

这里最重要的不是数值本身，而是你要先接受这种建模方式：

- 我不需要把所有真实细节都建精确
- 我只需要知道输入大概怎么影响系统
- 剩下那些不确定因素，交给扰动估计去处理

---

## 5. ESO 是什么

`ADRC` 里最核心的部件通常叫：

- `ESO`
- `Extended State Observer`
- 扩张状态观测器

“扩张”两个字的意思是：

- 不只估计原来系统里的位置和速度
- 还额外扩一个状态，用来估计总扰动

所以脚本里观测器状态是：

- `z1`：估计位置
- `z2`：估计速度
- `z3`：估计总扰动

这就是 `ADRC` 和很多普通状态观测器最不一样的地方：

- 它不是只想知道系统当前在哪、速度多大
- 它还想知道“现在到底有什么未知东西在推着系统跑偏”

---

## 6. 为什么 `z3` 能代表总扰动

先看真实系统：

`x_ddot = b0 * u + disturbance`

如果把右边的未知部分统一记成一个量 `f`，就可以写成：

`x_ddot = b0 * u + f`

这里的 `f` 就是总扰动。

于是 `ADRC` 的思路变成：

- `z1` 去估 `x`
- `z2` 去估 `x_dot`
- `z3` 去估 `f`

这样控制器一旦得到 `z3`，就能做一件很自然的事：

- 既然我知道系统里多出来一个未知推力
- 那我就在控制输入里把它减掉

所以 `z3` 的作用不是“看起来高级”，而是它给了控制器一个可以主动补偿未知项的抓手。

---

## 7. 逐段看 ESO 更新式

脚本里观测器部分是：

```python
y = x
e = z1 - y

z1 += dt * (z2 - beta1 * e)
z2 += dt * (z3 - beta2 * e)
z3 += dt * (-beta3 * e)
```

这里最先要看懂的是：

- `y` 是测量值
- 这个示例里假设能直接测到位置，所以 `y = x`
- `e = z1 - y` 是观测误差

也就是：

- 如果 `z1` 和真实测量差很多
- 说明当前观测器内部估计不对
- 那就要用误差 `e` 去强力拉回来

### 7.1 `z1` 的更新在做什么

```python
z1 += dt * (z2 - beta1 * e)
```

它表示：

- 位置估计 `z1` 会按当前估计速度 `z2` 往前走
- 同时如果和测量值有偏差，就用 `-beta1 * e` 把它拉回来

这和很多观测器的基本结构类似：

- 先按内部模型预测
- 再按测量误差纠正

### 7.2 `z2` 的更新在做什么

```python
z2 += dt * (z3 - beta2 * e)
```

这里很值得注意：

- `z2` 的变化不仅依赖误差修正
- 还依赖 `z3`

物理直觉是：

- 如果系统速度在变，通常背后是有加速度在推动
- 而这个“推动速度变化的未知影响”，就由 `z3` 来承担

所以 `z2` 并不是孤立更新的，它已经开始把“总扰动估计”纳入进来了。

### 7.3 `z3` 的更新在做什么

```python
z3 += dt * (-beta3 * e)
```

这一步可以理解成：

- 当观测器总觉得位置估计跟不上真实测量
- 它就会把这个长期解释不通的偏差，逐渐归因到“还有一个未知扰动没估进去”

于是：

- `z3` 会不断调整
- 直到观测器能用 `z1`、`z2`、`z3` 三者一起把系统行为解释通

这就是为什么 `z3` 最终会逐步逼近那个外扰大小附近。

---

## 8. 控制律为什么要写成现在这样

脚本里的控制律是：

```python
u0 = -kp * z1 - kd * z2
u = (u0 - z3) / b0
```

这两步要分开理解。

### 8.1 先看 `u0`

```python
u0 = -kp * z1 - kd * z2
```

这就是一个最普通的 `PD` 思路：

- 位置偏了，就拉回去
- 速度太大，也压一压

如果系统没有扰动，这一层已经能工作。

### 8.2 再看为什么要减 `z3`

前面我们把真实系统写成：

`x_ddot = b0 * u + total_disturbance`

如果现在观测器告诉你：

- 总扰动大约就是 `z3`

那么最自然的补偿方式就是：

- 在输入里先把这个扰动抵消掉

所以会写成：

`u = (u0 - z3) / b0`

如果把它代回系统：

```text
x_ddot = b0 * [(u0 - z3) / b0] + disturbance
       = u0 - z3 + disturbance
```

当 `z3 ≈ disturbance` 时，就变成：

```text
x_ddot ≈ u0
```

也就是说：

- 原本难处理的未知外扰，被补偿掉了
- 系统就重新接近你理想里那个“好控制的对象”

这就是 `ADRC` 最核心的控制逻辑。

### 8.3 为什么还要除以 `b0`

因为系统里输入 `u` 不是直接等于加速度，而是：

`x_ddot = b0 * u + ...`

所以如果你希望控制器产生的“等效加速度作用”是某个目标值，就必须除以 `b0`，把尺度换回去。

你可以把 `b0` 理解成：

- 输入到加速度之间的名义放大倍数

## 9. 主循环到底在做什么

主循环每一步都分成三层：

1. 读传感器输出
2. 更新 `ESO`
3. 根据估计状态和估计扰动计算控制输入
4. 再让真实系统前进一步

按代码顺序看会更清楚。

### 9.1 测量位置

```python
y = x
```

这个示例里没有额外测量噪声，先假设：

- 传感器可以直接测到位置

这样做是为了先把 `ADRC` 主线讲清楚，不让噪声问题干扰理解。

### 9.2 用位置误差更新 `ESO`

```python
e = z1 - y
z1 += dt * (z2 - beta1 * e)
z2 += dt * (z3 - beta2 * e)
z3 += dt * (-beta3 * e)
```

这一步是在同时更新：

- 位置估计
- 速度估计
- 总扰动估计

### 9.3 用估计状态做 `PD`

```python
u0 = -kp * z1 - kd * z2
```

这说明 `ADRC` 并不一定要求一个非常复杂的控制律。  
很多时候：

- 一个相对简单的反馈器
- 再加上扰动补偿

就已经很有力量。

### 9.4 用 `z3` 做扰动补偿

```python
u = (u0 - z3) / b0
u = float(np.clip(u, -0.25, 0.25))
```

这里先补偿，再限幅。

限幅的意义和前面几个示例一样：

- 模拟真实执行器能力有限
- 避免控制量无限大

### 9.5 让真实系统前进一步

```python
a = b0 * u + disturbance
v += a * dt
x += v * dt
```

这部分表示真实世界里确实还有外扰存在。  
`ADRC` 不是让扰动物理上消失，而是：

- 通过估计和补偿，让控制器“看起来像是把它抵消掉了”

## 10. `beta1`、`beta2`、`beta3` 怎么理解

这三个量是 `ESO` 的观测器增益。

最基础的直觉是：

- 增益大：估计更快，但更容易敏感
- 增益小：估计更稳，但跟踪更慢

更具体一点：

- `beta1`
  - 主要影响位置估计纠偏速度
- `beta2`
  - 主要影响速度估计纠偏强度
- `beta3`
  - 主要影响总扰动估计速度

如果这些增益太小，常见现象是：

- `z3` 跟不上扰动
- 补偿不及时
- 系统回零慢，甚至存在明显偏差

如果这些增益太大，真实系统里常见风险是：

- 对测量噪声更敏感
- 控制量抖动加重

这个示例没有显式测量噪声，所以你暂时不会感受到全部代价，但这个权衡以后非常重要。

## 11. 运行时应该重点看什么

运行：

```bash
python 06_helloworld_adrc/hello_adrc.py
```

输出里重点看这些量：

- `x`
  - 位置是否逐步回到 0
- `v`
  - 速度是否被压下来
- `z1`
  - 是否能跟住位置
- `z2`
  - 是否能反映速度变化趋势
- `z3`
  - 是否逐渐收敛到某个稳定值
- `u`
  - 是否长期打满限幅

正常情况下你通常会看到：

- `x` 被逐步拉回中心
- `z1` 跟位置比较接近
- `z3` 会从 0 开始逐步建立起来
- 一旦 `z3` 估得差不多，位置回零会明显更稳

## 12. 为什么它和普通 PD 的感觉不一样

如果你只用普通 `PD` 去控一个带恒定外扰的系统，经常会出现：

- 需要很大的增益才压得住
- 存在稳态偏差
- 对模型误差比较敏感

而 `ADRC` 想做的是：

- 把未知外扰单独估出来
- 不让 `PD` 自己硬扛全部误差

所以它给人的感觉通常是：

- 控制器本体不一定特别复杂
- 但因为多了一层扰动估计，整体更“抗偏差”

这也是为什么很多人第一次接触 `ADRC` 时，会觉得它不像传统控制那样只盯着误差，而是在主动“清理战场”。

## 13. 你可以立刻做的几个实验

### 实验 1：把外扰调大

把：

```python
disturbance = 1.5
```

改大，比如改成：

```python
disturbance = 3.0
```

观察：

- `z3` 是否还能跟住
- `x` 是否仍能回零
- 控制量 `u` 是否更容易撞到限幅

### 实验 2：把 `beta3` 调小

比如改成：

```python
beta3 = 2000.0
```

观察：

- `z3` 建立速度是否明显变慢
- 系统前期偏差是否更大

### 实验 3：把 `kp`、`kd` 调小

比如改成：

```python
kp = 20.0
kd = 6.0
```

观察：

- 即使有扰动补偿，反馈太弱时会发生什么

### 实验 4：暂时去掉扰动补偿

把：

```python
u = (u0 - z3) / b0
```

临时改成：

```python
u = u0 / b0
```

观察：

- 没有 `z3` 补偿时，系统是否更容易留下偏差

这个实验很有价值，因为它能直接让你看到：

- `ADRC` 的关键不是只有 `PD`
- 而是 `ESO + 扰动补偿`

## 14. 这份示例故意省略了什么

为了让主线清楚，这份示例省略了很多真实工程细节：

- 没有测量噪声
- 没有更复杂的非线性项
- 没有更严格的参数整定方法
- 没有离散实现细节分析
- 没有 2D 耦合

所以它不是完整工程版 `ADRC`，而是一个用最小代码讲清楚核心结构的 hello world。

先把这版吃透，最重要的是抓住：

- 为什么要引入 `z3`
- `ESO` 到底在干什么
- 扰动补偿为什么有效

## 15. 对未来平台的意义

如果以后你做实体 2D 平台，很可能会遇到：

- 桌面轻微不平
- 机械装配偏差
- 摩擦不一致
- 电机零偏
- 模型参数变化

这些都很适合用 `ADRC` 或类似的扰动估计补偿思路处理。

所以 `ADRC` 的价值不在于：

- 它一定全面优于 `LQR` 或 `MPC`

而在于：

- 当模型不够准、未知影响很多时，它给了你一条非常实用的工程路线

## 16. 小结

这个目录最值得你带走的是下面几件事：

- `ADRC` 全名是 `Active Disturbance Rejection Control`
- 它的核心不是“把模型建得更复杂”，而是“把未知项估出来再补偿”
- `ESO` 会同时估计位置、速度和总扰动
- `z3` 是这份方法的关键，因为它给了控制器主动补偿未知影响的能力
- 这份 hello world 的主线是：`ESO -> 估扰 -> 补偿 -> 回零`
