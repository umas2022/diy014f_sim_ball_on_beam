# Kalman Filter Hello World

这个目录先不强调“控制器升级”，而强调“状态估计”。  
你现在已经做过 `PID` 和 `LQR`，下一步最需要补上的能力，是：

- 只有位置测量时，怎么估计速度
- 测量有噪声时，怎么得到更平滑、更可信的状态

这正是 `Kalman Filter` 的典型用途。

---

## 1. 这个脚本到底在解决什么问题

脚本 `hello_kalman.py` 只做一件很基础但很关键的事：

- 真实系统里有位置 `x_true` 和速度 `v_true`
- 传感器只能测到带噪声的位置 `x_meas`
- 滤波器要根据“模型 + 测量”去实时估计：
  - `x_hat`：估计位置
  - `v_hat`：估计速度

换句话说，这不是在直接研究“怎么控得更好”，而是在研究：

- 我明明没测到速度，能不能把速度估出来
- 我测到的位置很抖，能不能估出一个更接近真实值的位置

这件事对后面几乎所有模型驱动控制都很关键，因为：

- `LQR` 默认你知道完整状态
- `LQG` 本质上就是 `LQR + Kalman`
- `MPC` 也通常更希望拿到完整状态

如果状态不完整，控制器再漂亮，落地时也会很别扭。

---

## 2. Kalman 里说的“模型”到底是什么

这里的“模型”很容易和机器学习里的“模型文件”混在一起，所以值得单独讲清楚。

在 Kalman Filter 里，所谓模型，不是 `.pt` 那种“训练好的网络参数”，而是：

- 系统怎么运动的数学描述
- 传感器怎么观测系统的数学描述

也就是说，Kalman 里的模型通常分成两部分。

### 2.5.1 状态转移模型

它回答的问题是：

- “如果我知道这一刻的状态，再给定输入，下一刻大概会变成什么样？”

本例里就是：

- `x_next = x + v * dt + 0.5 * a * dt^2`
- `v_next = v + a * dt`

写成矩阵，就是：

`x(k+1) = F x(k) + B u(k) + w(k)`

这里：

- `F` 描述状态自己怎么演化
- `B` 描述输入怎么推动状态变化
- `w(k)` 表示模型不可能完美，所以留一个过程噪声

### 2.5.2 测量模型

它回答的问题是：

- “传感器到底能从真实状态里看到什么？”

本例里传感器只能测位置，所以：

- `z = x + 测量噪声`

写成矩阵，就是：

`z(k) = H x(k) + v(k)`

这里：

- `H` 负责从状态里“取出”传感器能看到的那一部分
- `v(k)` 表示测量噪声

所以，本例的模型并不神秘，本质上就是两句话：

- 系统怎样从这一刻走到下一刻
- 传感器怎样从状态里读出测量值

### 2.5.3 它和 LQR 里的模型是什么关系

你前面在 LQR 里看到的模型，本质上和这里是同一类东西：

- 都是在描述系统动力学
- 都可以写成状态空间形式
- 都是“基于物理规律得到的数学方程”

区别只在用途：

- LQR 里的模型，主要用来设计控制律 `u = -Kx`
- Kalman 里的模型，主要用来预测状态并和测量融合

所以你可以把它理解成：

- LQR 用模型来“算怎么控”
- Kalman 用模型来“算系统现在大概在哪、速度多大”

### 2.5.4 它和机器学习里的“模型”有什么区别

在机器学习里，大家常说“模型”，通常指的是：

- 一个神经网络结构
- 一组训练好的参数
- 比如 `.pt`、`.pth` 这类权重文件

这类模型更像是：

- 从数据里学出来的映射关系

而 Kalman 这里的模型更像是：

- 一份物理规律说明书

它通常不是训练出来的，而是：

- 根据牛顿运动定律
- 根据运动学公式
- 根据传感器工作方式
- 由人先写出来的

### 2.5.5 Kalman 到底在持续更新什么

还有一个很关键的点：

- 标准 Kalman Filter 并不是在持续优化 `F`、`B`、`H`
- 它持续更新的是当前状态估计 `x_hat` 和不确定性 `P`

也就是说，每一步真正被在线更新的是：

- “我现在认为位置是多少”
- “我现在认为速度是多少”
- “我对这个判断有多确定”

而不是：

- “我重新学到了一个新的系统动力学模型”

所以如果用一句话概括：

- Kalman 的模型通常是先给定的
- Kalman 在线优化的是状态估计，不是模型本身

## 3. 先建立一个直觉：Kalman Filter 不是“神奇平均器”

初学者最容易把它理解成“把噪声做平滑”。  
这个理解不完全错，但不够准确。

更准确地说，Kalman Filter 每一步都在做两件事：

1. **先根据模型预测下一步会在哪里**
2. **再根据传感器测量修正这个预测**

所以它不是只看测量值，也不是只看模型，而是在动态权衡：

- 如果模型更可信，就多信模型
- 如果测量更可信，就多信测量

这就是后面 `Q` 和 `R` 两个参数的物理意义。

## 4. 脚本里的状态、输入、测量分别是什么

这个例子使用一个 1D 的最小模型：

- 状态：`[x, v]`
  - `x` 是位置
  - `v` 是速度
- 输入：`a_cmd`
  - 你可以把它理解成“系统认为自己施加的加速度命令”
- 测量：`z`
  - 这里只有位置测量

对应的离散状态空间模型是：

`x(k+1) = F x(k) + B u(k) + w(k)`

`z(k) = H x(k) + v(k)`

其中：

- `F`：状态如何自然演化
- `B`：控制输入如何影响状态
- `H`：传感器能从状态里读出什么
- `w(k)`：过程噪声，表示模型不完美
- `v(k)`：测量噪声，表示传感器有误差

这个目录里只测位置，所以 `H` 只把位置那一维取出来。

## 5. 先看脚本前半段：模型和噪声是怎么定义的

脚本开头这一段：

```python
dt = 0.01
T = 6.0

disturbance = 0.15
measurement_sigma = 0.01
process_sigma = 0.02
```

可以这样理解：

- `dt`：采样周期，每 `0.01s` 更新一次
- `T`：总仿真时间 `6s`
- `disturbance`：真实系统里一直存在的小外扰
- `measurement_sigma`：位置测量噪声标准差
- `process_sigma`：过程噪声标准差

这里最重要的不是数值本身，而是你要建立这个概念：

- **真实系统不会完全等于模型**
- **传感器也不会给你完美真值**

Kalman Filter 就是为这种现实情况设计的。

## 6. 为什么 `F`、`B` 要写成现在这样

脚本里有这几行：

```python
F = np.array([
    [1.0, dt],
    [0.0, 1.0],
], dtype=float)

B = np.array([
    [0.5 * dt * dt],
    [dt],
], dtype=float)
```

这其实就是把最基础的运动学离散化：

- `x_next = x + v * dt + 0.5 * a * dt^2`
- `v_next = v + a * dt`

所以：

- `F` 负责描述“如果没有输入，状态自己怎么往下走”
- `B` 负责描述“加速度输入会怎么改变位置和速度”

如果你已经学过 `LQR`，这里可以把它看成一个更简单的状态空间模型。  
区别是这次重点不是设计反馈增益，而是用这个模型辅助估计状态。

## 7. 为什么 `H = [[1, 0]]`

脚本里：

```python
H = np.array([[1.0, 0.0]], dtype=float)
```

这表示传感器只能看到状态向量 `[x, v]` 里的第一项，也就是位置 `x`。

所以这里的设定就是：

- 位置可测
- 速度不可测

这正好对应你后面做球板/平台时很常见的情况：

- 相机能看到位置
- 速度通常不是直接测来的，而是估计出来的

## 8. `Q` 和 `R` 到底是什么

脚本里：

```python
Q = np.array([
    [1e-5, 0.0],
    [0.0, process_sigma ** 2],
], dtype=float)
R = np.array([[measurement_sigma ** 2]], dtype=float)
```

可以把它们先理解成“你对世界不确定性的描述”：

- `Q`：过程噪声协方差，表示你觉得模型有多不准
- `R`：测量噪声协方差，表示你觉得传感器有多吵

初学时只记住这个直觉就够了：

- `Q` 变大：更不信模型，更愿意让测量来纠正
- `R` 变大：更不信测量，更愿意按模型往前推

你以后调 Kalman，最先感受到的就是这两个参数带来的权衡。

## 9. 真值、测量值、估计值三者分别是什么

脚本里有三套“状态”概念，不要混：

### 8.1 真值

```python
x_true = 0.05
v_true = 0.0
```

这代表仿真器里的真实系统状态。  
现实里你通常拿不到这个真值，它只是为了让我们验证滤波器效果。

### 8.2 测量值

```python
z = x_true + rng.normal(0.0, measurement_sigma)
```

这表示传感器读到的值。  
它等于真实位置加上一点测量噪声，所以会抖。

### 8.3 估计值

```python
x_hat = np.array([[0.0], [0.0]], dtype=float)
P = np.eye(2) * 0.1
```

其中：

- `x_hat` 是滤波器当前对状态的估计
- `P` 是估计误差协方差，表示“我对当前估计有多确定”

`P` 大表示不太确定，`P` 小表示更有把握。

## 10. 先读懂代码里的符号：`@`、`*`、`.T`、`inv()`

在 Kalman 代码里，很多初学者第一次卡住不是公式本身，而是 Python 写法。

脚本里最常见的是这几个记号：

- `@`
- `*`
- `.T`
- `np.linalg.inv(...)`

### 9.1 `@` 是矩阵乘法

在 `numpy` 里：

- `A @ x` 表示矩阵乘法
- 不是逐元素相乘

比如：

```python
F = np.array([
    [1.0, dt],
    [0.0, 1.0],
])

x_hat = np.array([
    [x],
    [v],
])
```

那么：

```python
F @ x_hat
```

算出来的是：

```text
[ 1*x + dt*v ]
[ 0*x + 1*v  ]
```

也就是：

- 第一行：预测下一步位置
- 第二行：预测下一步速度

所以 `@` 本质上是在做“按状态方程把各项组合起来”。

### 9.2 `*` 在这里通常是标量乘法

比如：

```python
B * a_cmd
```

这里 `a_cmd` 是一个标量，所以它表示：

- 把输入矩阵 `B` 的每一项都乘上这个加速度命令

因为 `B` 本来就在描述：

- 输入会以多大权重影响每个状态

所以 `B * a_cmd` 就是在算：

- 这次输入 `a_cmd` 实际给状态带来多少变化

### 9.3 `.T` 表示转置

比如：

```python
H.T
F.T
```

`.T` 表示矩阵转置，也就是：

- 行变列
- 列变行

例如：

```python
H = [[1, 0]]
H.T = [[1],
       [0]]
```

为什么 Kalman 里经常要转置？

因为协方差传播和增益计算里，矩阵维度必须对上。  
更直觉地说：

- `H` 是“从状态空间映射到测量空间”
- `H.T` 则可以理解成“把测量空间的信息再映射回状态空间”

### 9.4 `inv()` 表示求逆

```python
np.linalg.inv(S)
```

表示求矩阵 `S` 的逆矩阵。

在这个例子里，`S` 其实只有 `1x1`，所以它几乎就等于“取倒数”。

这里写成矩阵逆，是因为 Kalman Filter 的标准公式本来就是面向更一般的多维系统。  
以后如果你同时测位置和速度，或者测更多维状态，`S` 就不一定是 `1x1` 了。

### 9.5 为什么 `@` 在 Kalman 里这么常见

因为 Kalman Filter 本来就是状态空间方法。

它每一步都在处理这种事情：

- 当前状态怎样变成下一时刻状态
- 当前不确定性怎样传播到下一时刻
- 测量误差怎样反推回状态修正

这些都天然适合写成矩阵乘法，所以代码里会频繁出现 `@`。

如果你先记一句最重要的话：

- `@` 不是神秘语法，它只是“按线性代数规则把状态和模型连起来”

后面读公式就会轻松很多。

## 11. 主循环到底在做什么

主循环里每一步都分成三层逻辑：

1. 真实系统往前走一步
2. 传感器给一个带噪声的位置测量
3. Kalman Filter 做一次预测和更新

下面按脚本顺序拆开看。

### 10.1 先让真实系统运行

```python
a_cmd = -0.8 * x_true - 0.3 * v_true

process_noise = rng.normal(0.0, process_sigma)
a_true = a_cmd + disturbance + process_noise
v_true += a_true * dt
x_true += v_true * dt
```

这里不是在强调控制器设计，而只是为了让系统“动起来”，这样滤波器才有东西可估。

- `a_cmd` 是一个很简单的开环/弱反馈式激励
- `disturbance` 是恒定外扰
- `process_noise` 是随机过程噪声
- `a_true` 是真实系统实际受到的加速度

注意这个细节很重要：

- 滤波器在预测时只知道 `a_cmd`
- 但真实系统实际经历的是 `a_true`

两者之间有偏差，正是现实世界里“模型不完美”的来源。

### 10.2 再生成传感器测量

```python
z = x_true + rng.normal(0.0, measurement_sigma)
```

这一步表示传感器只给位置，而且有噪声。  
所以你看到的 `z` 并不等于 `x_true`。

### 10.3 Predict：先按模型往前推

```python
x_hat = F @ x_hat + B * a_cmd
P = F @ P @ F.T + Q
```

这一步回答的问题是：

- “如果先不看传感器，只按照模型推演，下一步状态应该是什么？”

两条式子的意义分别是：

- `x_hat = F @ x_hat + B * a_cmd`
  - 预测下一时刻的状态估计
- `P = F @ P @ F.T + Q`
  - 预测下一时刻的不确定性

#### 为什么状态预测要写成 `F @ x_hat + B * a_cmd`

因为预测阶段的核心问题是：

- “如果这一刻我暂时不看传感器，只根据上一刻估计值和已知输入，下一刻大概会到哪里？”

这里分成两部分：

- `F @ x_hat`
  - 表示“如果没有新的输入，旧状态自己会怎么自然演化”
- `B * a_cmd`
  - 表示“这次输入会额外给状态带来多少变化”

把两者加起来，就是：

- 旧状态自己往前走的部分
- 再加上输入推动出来的部分

这正是最基本的离散状态方程。

如果你把 `F` 和 `B` 展开来看：

```text
F @ x_hat =
[ x_hat + dt * v_hat ]
[ v_hat              ]

B * a_cmd =
[ 0.5 * dt^2 * a_cmd ]
[ dt * a_cmd         ]
```

所以合起来就是：

```text
x_hat_next = x_hat + dt * v_hat + 0.5 * dt^2 * a_cmd
v_hat_next = v_hat + dt * a_cmd
```

这和你熟悉的运动学公式完全一致。

为什么 `P` 会加上 `Q`？

- 因为你往前推一步以后，通常不会比之前更确定
- 模型本身有误差，所以不确定性会增加

#### 为什么协方差预测要写成 `F @ P @ F.T + Q`

这里不要急着背公式，先抓物理意思：

- `P` 表示你对当前估计误差的“不确定性描述”
- 当状态通过 `F` 往前传播时，这个不确定性也会一起传播

所以：

- `F @ P @ F.T`
  - 表示“旧的不确定性经过系统动力学传播到下一时刻后，变成什么样”
- `+ Q`
  - 表示“模型本身还有额外误差，所以再补上一层过程噪声”

也就是说，这一式并不是凭空发明的，而是在回答：

- 原来就有的不确定性会被系统带到下一步
- 走这一步过程中还会新增一些模型误差

两者加起来，才是预测后的总不确定性。

### 10.4 Update：再用测量把估计拉回来

```python
y = np.array([[z]], dtype=float) - H @ x_hat
S = H @ P @ H.T + R
K = P @ H.T @ np.linalg.inv(S)
x_hat = x_hat + K @ y
P = (np.eye(2) - K @ H) @ P
```

这一段是 Kalman Filter 的核心。

#### `y`：创新 / 残差

`y = z - H @ x_hat`

它表示：

- 传感器看到的值
- 减去
- 你根据模型预测传感器应该看到的值

如果 `y` 很大，说明预测和测量差很多，就需要更多修正。

为什么要先算 `y`，而不是直接拿测量值替换预测值？

因为 Kalman Filter 不想粗暴地说：

- “传感器来了，那就全听传感器”

它真正想知道的是：

- “这次测量和我原本预测差了多少？”

这个“差值”就是 `y`。  
后面所有修正，本质上都是围绕这个差值展开的。

#### `S`：残差的不确定性

`S = H P H^T + R`

可以理解成：

- 当前预测不确定性
- 加上传感器测量不确定性

为什么要先算 `S`？

因为你不能只看“残差大不大”，还得看：

- 这次残差到底值不值得大惊小怪

举个直觉例子：

- 如果传感器本来就很吵，那残差大一点可能很正常
- 如果传感器很准，那同样大小的残差就说明预测可能真的偏了

`S` 就是在衡量：

- 这次残差出现得“正常不正常”
- 它到底有多可信

#### `K`：Kalman 增益

`K = P H^T S^-1`

这一步很关键。  
`K` 决定“这次修正时到底该信测量多少”。

你可以这样记：

- `K` 大：修正幅度大，更信测量
- `K` 小：修正幅度小，更信模型

为什么 `K` 要这样计算，而不是手工设一个固定比例？

因为 Kalman Filter 想自动完成一件事：

- 当预测很不确定、测量又比较可靠时，就多信测量
- 当预测很稳定、测量又比较吵时，就少信测量

而 `P`、`R`、`S` 正好就编码了这种信息。

所以 `K` 不是拍脑袋设出来的常数，而是：

- 根据当前这一刻“模型有多不确定、测量有多不确定”
- 实时算出来的最合适修正权重

#### 更新状态和协方差

`x_hat = x_hat + K @ y`

表示把预测值按残差方向修正。

`P = (I - K H) @ P`

表示融合了这次测量之后，通常会比刚预测完更确定，所以不确定性下降。

你可以把 Update 整体记成一句话：

- 先看预测和测量差多少
- 再判断这次差值有多可信
- 最后按一个合适权重把状态和不确定性一起修正

---

## 12. 为什么它能估计出速度

这是初学者最关心的问题之一：

- 明明没有速度传感器
- 为什么 `v_hat` 还能出现

原因是速度虽然没被直接测量，但它会影响位置随时间的变化。  
Kalman Filter 利用的是：

- 状态之间的动力学耦合
- 连续时刻的位置测量

如果系统模型写得合理，位置的变化趋势里就已经包含了速度信息。  
滤波器会把这些信息“倒推出”速度估计。

这也是为什么状态空间方法这么重要：  
它让不同状态之间的关系可以被系统性利用。

## 13. 运行时应该重点看什么

运行：

```bash
python 03_helloworld_kalman/hello_kalman.py
```

输出里重点看这几组量：

- `x_true`：真实位置
- `x_meas`：带噪声的位置测量
- `x_hat`：滤波器估计位置
- `v_true`：真实速度
- `v_hat`：滤波器估计速度

正常现象通常是：

- `x_meas` 比较抖
- `x_hat` 比 `x_meas` 更平滑，也更接近 `x_true`
- `v_hat` 不会完全等于 `v_true`，但会逐渐跟住趋势

如果 `x_hat` 很平滑但反应特别迟钝，常见原因是：

- `R` 设太小或太大导致权衡不合理
- `Q` 太保守，滤波器过度相信模型

## 14. 你可以立刻做的三个小实验

### 实验 1：把测量噪声变大

把：

```python
measurement_sigma = 0.01
```

改大，比如改成 `0.05`。

观察：

- `x_meas` 会更抖
- `x_hat` 仍然会尽量保持稳定

这样你会更直观看到滤波器的价值。

### 实验 2：把 `R` 调大

让滤波器更不信测量。

你会看到：

- `x_hat` 更平滑
- 但有时会更慢地贴近真实变化

### 实验 3：把 `Q` 调大

让滤波器承认“模型没那么准”。

你会看到：

- 滤波器更愿意跟着测量修正
- 响应可能更快，但也可能更敏感

这三个实验做完，你对 Kalman 的调参直觉会明显更清楚。

## 15. 它和你已经学过的 PID、LQR 是什么关系

- `PID`：主要解决“怎么根据误差直接出控制量”
- `LQR`：主要解决“如果完整状态已知，怎么最优反馈”
- `Kalman`：主要解决“如果状态不完整、测量有噪声，怎么估计状态”

所以它们不是互相替代，而是不同层次的问题。

你当前这条学习路线很合理：

1. 先理解 `PID`
2. 再理解 `LQR`
3. 然后补上 `Kalman`
4. 最后组合成 `LQG`

这样逻辑是连贯的。

## 16. 对未来 2 自由度平台的直接意义

未来上 2D 平台时，你很可能会变成：

- 状态：`[x, vx, y, vy]`
- 测量：相机给出 `(x, y)`
- 估计：通过滤波器补出 `(vx, vy)`

那时本质没有变，只是从 1D 变成 2D：

- 现在你学的是“只测 `x`，估计 `v`”
- 以后会变成“只测 `x, y`，估计 `vx, vy`”

所以这个 hello world 虽然小，但它是后面所有“估计 + 控制”链路的基础。
