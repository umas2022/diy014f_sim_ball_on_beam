# MPC Hello World

这里的 `MPC` 全名是 `Model Predictive Control`，也就是模型预测控制。

如果你前面已经看过 `LQR`、`Kalman`、`LQG`，那么到 `MPC` 这一步，最重要的变化不是“公式更复杂了”，而是控制器开始主动回答一个更工程化的问题：

- 我不只想让系统稳定
- 我还想提前看看未来几步会发生什么
- 并在输入、角度、速度等约束下，选一个更合适的控制动作

这也是为什么 `MPC` 在工业控制、自动驾驶、机器人轨迹跟踪里都很常见。

---

## 1. 先建立一个直觉：MPC 到底在做什么

`MPC` 的核心思想可以先压缩成 5 句话：

1. 用模型预测未来一小段时间会怎么发展
2. 把未来几步的状态偏差和控制代价一起写成优化目标
3. 在约束条件下求一串未来控制输入
4. 但当前时刻只执行第一步
5. 下一时刻重新预测、重新优化

这就是常说的：

- `receding horizon`
- `rolling horizon`
- 滚动优化
- 滚动时域控制

如果只用一句最朴素的话理解它，就是：

- `LQR` 更像“一次算出一个固定反馈律”
- `MPC` 更像“每一步都重新做一次短期规划”

---

## 2. 为什么它值得学

`MPC` 特别适合处理下面这些真实问题：

- 输入有限幅
- 杆角有限制
- 角速度有限制
- 不只想“稳住”，还想“按某种方式稳住”
- 希望提前避免未来会碰到的约束

对未来 2 自由度平台来说，这很重要，因为真实系统几乎一定会有：

- 平台最大倾角限制
- 电机饱和
- 输入变化率限制
- 某些区域不希望走得太激进

这些问题单靠“当前误差有多大”通常不够，需要“看未来再决定现在”。

---

## 3. 这个 hello world 做了哪些简化

这份脚本故意把问题压到了最小版本，避免一开始就被优化器和约束细节淹没。

它做了这些简化：

- 只做 1D，不做 2D 平台
- 状态只保留 `x` 和 `v`
- 控制输入直接看作小角度杆角命令 `theta_cmd`
- 使用线性离散模型
- 使用二次型代价函数
- 不调用外部优化器
- 约束只做一个最直观的角度限幅 `theta_limit`

所以这份脚本的重点不是“工业级 MPC 求解器怎么接”，而是先讲清楚：

- 预测窗口是什么
- 代价函数是什么
- 为什么每次只执行第一步
- 它和 `LQR` 到底差在哪

---

## 4. 这个脚本里的状态、输入和模型是什么

脚本里状态定义为：

- `x`：小球位置
- `v`：小球速度

控制输入定义为：

- `theta_cmd`：杆角命令

这里仍然沿用小角度近似的直觉：

- 杆稍微倾斜，小球就会沿杆方向加速
- 杆角越大，小球加速度越大
- 小球速度同时受到一点阻尼

所以这个脚本用的离散模型是：

```python
F = np.array([
    [1.0, dt],
    [0.0, 1.0 - damping * dt],
], dtype=float)

G = np.array([
    [0.0],
    [g * dt],
], dtype=float)
```

对应的状态方程可以读成：

`x(k+1) = F x(k) + G u(k)`

也就是：

```text
x_next = x + dt * v
v_next = (1 - damping * dt) * v + g * dt * theta_cmd
```

这两条式子的物理意义分别是：

- 位置的下一步，来自当前速度积分
- 速度的下一步，来自阻尼衰减和杆角带来的加速度

如果你前面看过 `LQR`，这里应该会有熟悉感，因为它仍然是状态空间模型。  
只是 `MPC` 接下来不会直接求一个无限时域最优反馈，而是会把“未来 horizon 步”一起拿来考虑。

---

## 5. `horizon` 是什么

脚本里：

```python
horizon = 15
```

它表示预测窗口长度，也就是：

- 控制器每次向前看 15 步

由于这里 `dt = 0.02`，所以对应的预测时间大约是：

- `15 * 0.02 = 0.30 s`

这意味着控制器不是只看“现在这一瞬间”，而是在想：

- 如果我接下来 0.3 秒内怎么控制，系统会往哪走？

这就是 `MPC` 和很多传统控制器最直观的区别。

---

## 6. 代价函数到底在罚什么

脚本里定义了：

```python
Q = np.diag([80.0, 8.0])
R = np.array([[1.0]])
Qf = np.diag([120.0, 12.0])
```

你可以把它们理解成：

- `Q`：预测窗口中间每一步的状态偏差代价
- `R`：控制输入代价
- `Qf`：预测窗口最后一步的终端代价

直觉上：

- `x` 权重大，表示不希望小球离中心太远
- `v` 也有代价，表示不希望小球冲得太快
- `R` 不为 0，表示不希望杆角命令太激进
- `Qf` 更大，表示不希望“前面看着还行，最后几步却收不回来”

如果把它写成概念上的目标，大概就是：

- 让未来几步的位置和速度尽量接近 0
- 同时不要为了回中而给出过大的控制动作

这和 `LQR` 的思路是一脉相承的，只是 `MPC` 是在有限预测时域内做这件事。

---

## 7. 这个脚本里的 MPC 是怎么求出来的

脚本没有调用通用优化器，而是走了最适合入门理解的这一版：

- 线性模型
- 二次代价
- 有限时域
- 用 Riccati 反向递推求每一步反馈增益

核心函数是：

```python
def finite_horizon_gain(F, G, Q, R, Qf, horizon):
    p = Qf.copy()
    gains = []
    for _ in range(horizon):
        k = np.linalg.inv(R + G.T @ p @ G) @ (G.T @ p @ F)
        gains.append(k)
        p = Q + F.T @ p @ (F - G @ k)
    gains.reverse()
    return gains
```

它的思路是：

1. 先假设预测窗口最后一步的代价是 `Qf`
2. 从最后一步往前倒推
3. 每倒推一步，就得到该步最优反馈增益 `k`
4. 最后得到整个预测窗口里每一步对应的一组增益

这其实就是有限时域最优控制的经典做法。

如果你把它翻译成直觉语言：

- 我先规定“走到预测窗口末尾时，也不能太离谱”
- 然后从后往前想，每一步最合理的控制该是什么

所以这里虽然标题写的是 `MPC`，但本质上你可以把它看成：

- 最小版有限时域线性二次 `MPC`

## 8. 为什么每次只执行第一步控制

主循环里这一句很关键：

```python
theta_cmd = float(-(gains[0] @ x)[0, 0])
```

意思是：

- 虽然内部规划了未来 `horizon` 步
- 但当前只执行第 1 步控制

为什么不能把未来 15 步一次性全执行完？

因为真实系统下一步状态一出来，情况就变了：

- 模型预测可能有误差
- 扰动可能变了
- 约束可能更接近了
- 当前状态已经不是 15 步前假设的那个状态

所以 `MPC` 的核心不是“规划很长一串然后死执行”，而是：

- 每一步都重新看当前状态
- 每一步都重新做一次短期最优决策

这也是它比“先离线算一整条输入序列再执行”更稳健的原因。

---

## 9. 这份脚本为什么看起来不像“真优化器”

这点值得说清楚，不然你容易误会。

很多人印象里的 `MPC` 是：

- 每一步都调用一个数值优化器
- 显式处理不等式约束
- 在线求一串最优控制量

而这份脚本没有这么做。  
它更像是：

- 用有限时域 Riccati 反向递推，先把最优反馈结构算出来
- 在线阶段只做矩阵乘法

所以它是一个很适合入门的 `MPC` 版本，但不是完整工业版求解器演示。

更具体地说，这份脚本里的角度限制：

```python
theta_cmd = float(np.clip(theta_cmd, -theta_limit, theta_limit))
```

是“优化之后再限幅”，而不是把约束直接写进优化问题里。

这意味着：

- 它已经让你体验到“约束很重要”
- 但还没有走到“真正显式约束优化”的完整形态

这个区分很重要。

---

## 10. 为什么说它是从 LQR 走向 MPC 的自然下一步

如果：

- 模型线性
- 代价二次
- 没有显式约束
- 预测时域越来越长

那么有限时域 `MPC` 会越来越接近 `LQR`。

所以它们不是互相对立的两套世界，而是连续的：

- `LQR`：无限时域、无约束、一次求出反馈律
- `MPC`：有限时域、可加约束、每一步重做短期规划

你可以把这份示例理解成：

- 从 `LQR` 走向“带预测、可带约束控制”的第一步

这也是为什么前面先学 `LQR`，再来看 `MPC` 会顺很多。  
因为你已经见过：

- 状态空间模型
- 二次型代价
- Riccati 递推

这里只是把“考虑未来多步”这件事进一步强化了。

## 11. 逐段读 `hello_mpc.py`

### 11.1 基本参数

```python
g = 9.81
dt = 0.02
T = 6.0
damping = 0.25
theta_limit = np.deg2rad(12)
horizon = 15
```

这里分别表示：

- `g`：重力加速度
- `dt`：采样周期
- `T`：总仿真时间
- `damping`：速度阻尼
- `theta_limit`：杆角命令限幅
- `horizon`：预测窗口长度

### 11.2 状态空间模型

```python
F = np.array([
    [1.0, dt],
    [0.0, 1.0 - damping * dt],
], dtype=float)

G = np.array([
    [0.0],
    [g * dt],
], dtype=float)
```

这部分就是告诉控制器：

- 如果当前状态是 `[x, v]`
- 给定一个杆角命令 `theta_cmd`
- 下一步状态会怎么变化

### 11.3 代价权重

```python
Q = np.diag([80.0, 8.0])
R = np.array([[1.0]])
Qf = np.diag([120.0, 12.0])
```

这部分告诉控制器：

- 哪些状态偏差更不能忍
- 控制动作有多“贵”
- 预测窗口末端有多重要

### 11.4 反向递推求增益

```python
gains = finite_horizon_gain(F, G, Q, R, Qf, horizon)
```

这一步是在预测窗口内，提前算出每一步的最优反馈增益。

### 11.5 每一步只执行第一步控制

```python
theta_cmd = float(-(gains[0] @ x)[0, 0])
theta_cmd = float(np.clip(theta_cmd, -theta_limit, theta_limit))
x = F @ x + G * theta_cmd
```

这 3 句对应 `MPC` 的完整闭环逻辑：

1. 根据当前状态算第 1 步最优控制
2. 做限幅，模拟真实约束
3. 用模型把系统推进一步

## 12. 运行时应该重点看什么

运行：

```bash
python 05_helloworld_mpc/hello_mpc.py
```

输出里重点看三件事：

- `x` 是否逐步回到 0
- `v` 是否被压住，没有一直冲来冲去
- `theta_cmd` 是否频繁撞到 `theta_limit`

如果表现正常，你通常会看到：

- 小球位置逐步回中
- 速度逐渐减小
- 杆角命令在早期比较积极，后期逐渐变小

如果 `theta_cmd` 长时间都顶在限幅边界，通常说明：

- 初始偏差较大
- 或者权重设置太激进
- 或者预测窗口太短，控制器只顾眼前

## 13. 为什么 `horizon`、`Q`、`R` 很重要

### 13.1 `horizon` 太短会怎样

如果预测窗口太短，控制器会更像“只顾眼前几步”：

- 可能反应很快
- 但不够有前瞻性
- 更容易出现短视决策

### 13.2 `horizon` 太长会怎样

如果预测窗口很长，控制器会更愿意考虑远期效果：

- 前瞻性更强
- 但计算量会增加

在这个小示例里计算量不敏感，但在真实系统里这是核心权衡之一。

### 13.3 `Q` 大会怎样

`Q` 变大表示更不希望状态偏离目标：

- 回中会更积极
- 控制通常也更激进

### 13.4 `R` 大会怎样

`R` 变大表示更不希望控制动作太大：

- 杆角命令更温和
- 收敛一般会更慢

这些调参直觉，其实和 `LQR` 很像。  
不同的是 `MPC` 多了一个非常关键的旋钮：

- 预测窗口 `horizon`

## 14. 你可以立刻做的几个实验

### 实验 1：把 `horizon` 调短

比如改成：

```python
horizon = 5
```

观察：

- 控制是否更短视
- `theta_cmd` 是否更容易激进
- 收敛是否变差

### 实验 2：把 `R` 调大

比如改成：

```python
R = np.array([[5.0]])
```

观察：

- 杆角命令是否明显变小
- 小球回中的速度是否变慢

### 实验 3：把 `theta_limit` 调小

比如改成：

```python
theta_limit = np.deg2rad(6)
```

观察：

- 约束更紧时，系统是否还能顺利回中
- 控制是否更容易“力不从心”

### 实验 4：把初始偏差调大

把：

```python
x = np.array([[0.05], [0.0]], dtype=float)
```

改成：

```python
x = np.array([[0.10], [0.0]], dtype=float)
```

观察：

- 初始偏差更大时，限幅是否更容易触发
- 预测控制在大偏差下会表现出什么变化

## 15. 这份示例故意省略了什么

为了让主线清楚，这份示例省略了很多真实工程里常见的部分：

- 没有状态估计器
- 没有显式约束优化器
- 没有输入变化率约束
- 没有终端约束
- 没有模型失配和外扰分析
- 没有 2D 耦合

所以它不是“完整工业 MPC”，而是：

- 一个把最核心概念先讲清楚的 hello world

先把这个版本吃透，再看真正基于 `QP` 求解器的约束 `MPC`，会容易很多。

## 16. 对未来 2 自由度平台的意义

未来如果你做平面小球平台，`MPC` 的吸引力会更明显，因为它天然适合扩展成：

- 同时控制 `x / y`
- 同时考虑两个方向耦合
- 加入平台角度约束
- 加入角速度约束
- 加入输入变化率约束
- 做轨迹跟踪而不只是定点稳定

也就是说，`MPC` 的真正价值不只是“又一种控制器”，而是：

- 它给了你一个系统化处理预测和约束的框架

## 17. 学完这个目录后，下一步怎么看

如果你现在刚看到 `MPC`，建议这样理解它在整条学习路径里的位置：

1. `PID`
   - 学会最直接的误差反馈控制
2. `LQR`
   - 学会用模型和代价函数统一设计状态反馈
3. `Kalman / LQG`
   - 学会在状态不完整、测量有噪声时怎么估计和控制
4. `MPC`
   - 学会把“未来预测”和“约束”正式放进控制器里

如果你把这条线串起来，就会更清楚地看到：

- `PID` 更偏经验反馈
- `LQR` 更偏模型驱动最优反馈
- `MPC` 更偏在线短期规划

---

## 18. 小结

这个目录最值得你带走的，不是某一条公式，而是这几个核心认识：

- `MPC` 全名是 `Model Predictive Control`
- 它的本质是“预测未来 + 优化未来 + 只执行第一步”
- 它和 `LQR` 不是割裂的，而是自然延伸关系
- 它特别适合处理约束问题
- 这份脚本是最小版入门示例，重点在于建立结构直觉，而不是展示完整工业求解器
